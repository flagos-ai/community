# 决策操作手册

[English](decision-guide.md) | [中文](decision-guide_CN.md)

本文档是 [GOVERNANCE.md](../GOVERNANCE.md) 的操作细则，告诉你每种决策类型的**具体执行步骤**。

---

## 一、Lazy Consensus（默认决策方式）

**适用场景**: 日常 PR 审批、FEP 推进、SIG 内部常规决策。

### 操作步骤

**发起人：**

1. 在 GitHub PR 描述中标注 `**Decision**: <决策内容>`
2. 在 PR comment 中 @mention 相关 Approver 和 SIG Chair
3. 等待 **72 小时**（从 PR 创建或最后一次实质性修改算起）
4. 若 72 小时内无 Approver 反对、无阻塞性 review comment → **决策通过**
5. 在 PR 合并前，发起人在 comment 中确认：`决策通过 (lazy consensus, 72h 无反对)`

**Approver：**

1. 收到 @mention 后，在 72 小时内给出立场
2. 同意 → LGTM 或 approve
3. 需要修改 → 留 review comment，**明确标注是否阻塞**：`非阻塞：建议 xxx` vs `阻塞：需先解决 xxx`
4. 反对 → 在 PR comment 中明确写 `反对：<理由>`，并提供替代方案

**Chair：**

1. 确保相关 Approver 在 72 小时内注意到了提案
2. 若 Approver 未响应，Chair 负责提醒
3. 提醒后 48h 仍无响应 → 该 Approver 视为弃权，不影响计时
4. 若所有 Approver 均无响应 → Chair 可自行决定或升级 TSC
5. 若出现反对，Chair 组织讨论，尝试达成共识
6. 48 小时内无法达成共识 → 升级为投票决策

### 时间计算

```
起点: PR 创建时间 或 最后一次实质性修改时间
终点: 起点 + 72h
排除: 周末和法定假日不计入（实际按日历日计算，但 Approver 可申请延期至下一工作日）
```

### 状态标注

PR 中使用 label 辅助追踪：

| Label | 含义 |
|-------|------|
| `needs-decision` | 提案需要 Approver 决策 |
| `lazy-consensus` | 正在等待 72h 到期 |
| `under-discussion` | 讨论中，计时暂停 |
| `decision-reached` | 决策已达成 |

## 二、投票决策

**适用场景**: 新 SIG 创建/关停、版本 Go/No-Go、TSC 成员变更、角色晋升、以及 lazy consensus 无法达成共识时的升级。

### 操作步骤

**发起人：**

1. 在对应的 GitHub Issue 或 PR 中发起投票
2. 明确写出投票选项（通常为 "同意 / 反对 / 弃权"）
3. 设定投票截止时间（建议 ≥72h）
4. @mention 所有有投票权的成员
5. 投票截止后统计票数，宣布结果

> **利益冲突声明**：投票人如涉及利益冲突（详见 [GOVERNANCE.md](../GOVERNANCE.md) 利益冲突条款），应在投票 comment 中主动声明。声明后仍可正常投票，不强制回避。

**投票模板：**

```markdown
## 投票: <议题>

**发起人**: @github-id
**截止**: YYYY-MM-DD HH:MM (UTC+8)
**法定人数**: ≥50% (N/N)
**通过条件**: 2/3 多数

**选项**:
- [ ] 同意
- [ ] 反对
- [ ] 弃权

**背景**: (链接到提案文档/PR)

请有投票权的成员在此 Issue 下以 comment 形式投票。
```

### 计票规则

| 规则 | 说明 |
|------|------|
| **法定人数** | 参与投票的人数必须 ≥ 有投票权总人数的 50%，否则投票无效 |
| **通过条件** | 同意票 / (同意票 + 反对票) ≥ 2/3。弃权票不计入分母 |
| **平票** | 若同意和反对票数相等，Chair 有额外一票决定权 |
| **未投票** | 在截止前未投票视为弃权 |
| **无效投票** | 参与人数未达法定人数 → 投票无效。Chair 在一周内重新发起投票，可调整投票人范围或延长投票期。连续两次无效 → Chair 或 TSC 直接决定 |

### 投票结果记录

投票结果必须记录在 Issue/PR 中，格式：

```markdown
## 投票结果

**同意**: @a, @b, @c (3)
**反对**: @d (1)
**弃权**: @e (1)
**未投票**: @f (1)

**结果**: 通过 (3/4 = 75%, 达到 2/3)

**后续**: <决议执行步骤>
```

## 三、反对处理流程

当 lazy consensus 或投票中出现反对时：

```
反对提出
  │
  ├→ 反对者必须提供：1) 反对理由 2) 替代方案建议
  │
  ├→ Chair 组织讨论（48h 内）
  │     │
  │     ├→ 达成共识 → 修改提案，重新发起决策
  │     │
  │     ├→ 反对者撤回 → 继续原流程
  │     │
  │     └→ 无法达成共识 → 升级投票
  │           │
  │           └→ 仍无法通过 → 升级到上一级 (SIG → TSC)
  │                 │
  │                 └→ TSC 仍无法通过 → 提案搁置，60 天冷却期后可重新提出
  │                       │
  │                       └→ 紧急事项 → TSC Chair 临时决定（需 ≥1 名其他 TSC 成员联署），72h 内追溯投票；仍不通过则决议撤销
  │
  └→ 反对的理由和替代方案必须记录在决策文档中
```

## 四、紧急决策

**适用场景**: 安全事故、CI 全线阻塞、发布阻塞等需要立即行动的情况。

### 操作步骤

1. **TSC Chair + ≥1 名其他 TSC 成员联署**方可做临时决定。Chair 缺席时，任意 2 名 TSC 成员联署
2. 决定执行后 **24 小时内**在 GitHub 创建一个 Issue 记录：
   - 发生了什么
   - 做了什么决定
   - 为什么不能等正常流程
   - 联署人
3. **72 小时内** TSC 追溯投票确认
4. 若追溯投票未通过 → 回滚紧急决策、修复影响、反思流程漏洞

## 五、会议决策

### TSC 例会中的决策

1. 议程中明确标注哪些议题需要决策（`[决策] 议题标题`）
2. 会上讨论后，Chair 发起投票或确认 lazy consensus
3. 决议写入会议纪要，标注 `[决议]` 前缀
4. 纪要发布后 72 小时内，未参会的 TSC 成员可提出异议
5. 无异议 → 决议生效

### SIG 例会中的决策

参照 TSC 流程，但仅影响本 SIG 范围。涉及跨 SIG 的决策需升级到 TSC。

## 六、决策类型速查表

| 决策内容 | 方式 | 投票人 | 通过条件 |
|----------|------|--------|----------|
| 合入普通 PR | Lazy consensus | SIG Approver | 72h 无反对 |
| 合入 FEP (Provisional → Implementable) | Lazy consensus | SIG Approver | 72h 无反对 |
| 创建新 SIG | 投票 | TSC | 2/3 |
| 关停/合并 SIG | 投票 | TSC | 2/3 |
| 版本 Go/No-Go | 投票 | TSC | 2/3 |
| TSC 成员变更 | 投票 | TSC (成熟期由社区选举) | 2/3 |
| Reviewer 晋升 Approver | 投票 | 所属 SIG Approver | 2/3 |
| Approver 晋升 Tech Lead | TSC 批准 | TSC | Lazy consensus |
| Tech Lead 晋升 Chair | TSC 批准 | TSC | Lazy consensus |
| 紧急决策 | Chair + ≥1 TSC 成员联署 → 追溯 | TSC | 72h 内追溯通过 |
| GOVERNANCE.md 修改 | 投票 | TSC | 2/3 |
| CODE_OF_CONDUCT 修改 | 投票 | TSC | 2/3 |

## 七、决策记录规范

每个正式决策必须包含以下要素：

```markdown
### 决策记录: #NNN

- **日期**: YYYY-MM-DD
- **决策方式**: Lazy consensus / 投票 (x/x 通过)
- **决策内容**: <一句话描述>
- **决策人**: @github-id, @github-id
- **反对意见**: (如有，记录反对理由和替代方案)
- **后续行动**: [ ] @someone: xxx
```

决策记录存放在相关文档的同一 PR 中，或会议纪要中。
